SoSR-35

1 Scope

The management of electronic transport regulations (METR) provides trustworthy, authoritative, machine-interpretable, transport-related rules for using the road network. 

The scope of METR includes both rules that are relatively static (e.g., static speed limits) as well as those that are dynamic (e.g., variable speed limits, lane closures at incident scenes). However, where appropriate, METR is expected to incorporate existing standards (e.g., ISO/TS 19091 for signalized intersections).

This document defines the end-to-end requirements for entire METR system of systems (SoS), which is called the METR network. The requirements are provided in a format that is consistent with the guidance provided by ISO 29148 [2] for a system requirements document.

The operational concept for METR is provided in ISO 24315-2.

5 Overview
SoSR-2

5.1 System purpose

The Management of Electronic Transport Regulations (METR) intends to provide trustworthy transport-related rules in a machine-interpretable format to transport users. 
SoSR-3

5.2 System scope

The METR network is designed to provide electronic systems with the rules that travellers are required to follow. Provision of this rule information can be used directly by the receiving electronic system and can also be displayed to travellers to confirm which rules are currently relevant. To achieve this, the METR network provides a path and process to distribute the rules defined by rule-makers to METR users (e.g., travellers), who are required to comply with the rules.

Some rules are dependent upon supporting data, which can provide parameters related to a rule (e.g., the current speed limit for a variable speed limit system) or parameters that activate rules (e.g., the presence of rain might result in a lower speed limit). The METR network is responsible for delivering rules entered by a translator to (typically mobile) user systems. The user system is responsible for interpreting these rules along with supporting data from other sources (e.g., visually observed rules, current traffic conditions, and other factors) and providing interpreted results to the user, as needed. The METR network also includes the ability to report observations from the field back to the central office to ensure that the electronic data store is consistent with the rules posted in the field.

5.3.2 User characteristics
SoSR-180

5.3.2.1 General

The METR network interacts with translator agents, system operators, and METR users.
SoSR-181

5.3.2.2 Translator agents

Translator agents are responsible for translating rules, as defined by rule-makers, from their native language into an interoperable electronic format. As the rules are entered, they are electronically signed so that there is non-repudiation as to who entered the information. There are potentially many translator agents; different jurisdictional entities are likely to use different translator agents. 
SoSR-375

5.3.2.3 System operators

System operators are responsible for operating and maintaining components of the METR network.
SoSR-182

5.3.2.4 METR users

METR users are responsible for being aware of relevant rules and complying with them. They can use the METR network to obtain these rules and to supplement this information with other relevant observations. Every traveller can be considered a distinct METR user; in addition, driving automation systems and ancillary users are also distinct METR users.

SoSR-183

5.4.1 Assumptions
SoSR-188

5.4.1.1 Internet-based communications

The METR network will likely use the Internet and internet-enabled mobile telecommunication services to distribute pre-announced rules.
SoSR-207

5.4.1.2 Information delivery

The METR network should be designed and operated to ensure that each METR user system can have a high level of confidence that it will receive each rule and each piece of supporting data prior to the rule becoming relevant. When distributing emergent rules, this may require redundant systems due to the typical localized nature and short time horizon that exists to acquire the rule.
SoSR-410 EXAMPLE:

A jurisdiction uses a wide-area information disseminator system (WAIDS, e.g., FM broadcast subcarrier, satellite) to distribute emergent rules to all travellers. In addition, for safety-critical rules, the jurisdiction deploys local beacons as a secondary communication path in case the WAIDS is unable to deliver the information.
SoSR-411 NOTE:

The probability that a user system can access an emergent rule is impacted by the number of distribution points reachable, the availability of the rule through each distribution point (e.g., the number of times the rule is broadcast while the user system is in range), the transmission error rates involved, the resiliency of the distribution point, and other factors.

5.5 Traceability conventions

Within the systems engineering process, every managed item (e.g., user need, requirement, design element, etc.) should exist for a reason. At the genesis of the process, the reasons are documented through stakeholder consensus and are generally defined through a justification of a user need. All subsequent managed items are justified through traceability to managed items previously defined (e.g., design elements trace to requirement, which trace to user needs). Traces are therefore defined as being "from" the new managed item and "to" the previously defined managed item. Figure 3 depicts the traceability rules that apply to managed items defined within METR documents.  

From the perspective of developing the content of the ISO 24315 series, the traceability is normative in that traceability rules must be followed (e.g., each SoSR requirement must fulfill one or more user needs or constraints) while producing each document.  However, from the perspective of a reader of the ISO 24315 series, traceability is informative (i.e., non-normative) and only provides the justification as to why these requirements exist (e.g., system requirements fulfilling SoSR requirements) and what features would be impacted by a failure to meet certain requirements.

8.2 Systems within the SoS

The METR network provides METR users with trustworthy, authoritative rules that are currently active based on current location, time, and other conditions.  There are four main functions that are collectively performed by the functional roles with the SoS boundary.
SoSR-936

  • Translate jurisdictional rules into trustworthy electronic format and verify that these electronic rules correctly reflect corresponding physical implementations (e.g., traffic control devices).  These functions are performed by any system that assumes the translator role
SoSR-937

  • Distribute verified and digitally signed electronic rules to users based on their requested criteria, or as necessary for emergent conditions.  These functions are performed by any system that assumes the collector and/or disseminator roles.
SoSR-938

  • Ensure that required electronic rules are made available for end users (e.g., driving automation systems) to use in conducting driving operations in a lawful manner and to capture, track and report any discrepancies; and if so configured, discover and report rules.  These functions are performed by any system that assumes to role of the receiver, discovery reporter, and/or discrepancy reporter roles
SoSR-939

  • Collect and process discrepancies reported by end users, and report consolidated discrepancies to the associated translators.  These functions are performed by any system that assumes the discrepancy handler role.

8.3 Physical objects
SoSR-1029

8.3.1 Connected vehicle (CV) roadside equipment (RSE)

The CV RSE represents the roadside devices that are used to send messages to, and receive messages from, nearby vehicles and other mobile devices using wireless communications technologies. This is an existing ARC-IT ITS component; for a full description, please refer to ARC-IT.  

Within the context of METR, the CV RSE is used to distribute localized METR information, especially emergent rules. CV RSEs are also a primary source for infrastructure-sourced supporting data (e.g., signal timing information, infrastructure-based sensor information).
SoSR-1045 NOTE:

CV RSEs typically use wireless technologies with a transmission range of a few hundred metres.
SoSR-1030

8.3.2 Discrepancy contact system (DCS)

The DCS represents any external system that a METR regulation system might need to notify about the existence of a discrepancy report. For example, the original equipment manufacturer (OEM) for a vehicle model that repeatedly incorrectly reports discrepancies should be notified of the problem in its discrepancy reporting logic. 
SoSR-1207

8.3.3 ITS roadway equipment (IRE)

The IRE represents roadside devices that monitor and/or control conditions along the transportation network. This is an existing ARC-IT ITS component; for a full description, refer to ARC-IT.

Within the context of METR, the IRE is used to gather sensor data (e.g., traffic information, weather information) that can be processed and provided to METR users as supporting data.
SoSR-1031

8.3.4 Maintenance and construction management centre (M&CMC)

The M&CMC monitors and manages construction and maintenance activities on transport facilities.  This is an existing ARC-IT ITS component; for a full description, please refer to ARC-IT.  

Within the scope of METR, the M&CMC is responsible for planning, installing, and maintaining all traffic control devices (TCD) that correspond to jurisdictional rules, as well as detectors and sensors that provide supporting data.  Furthermore, the M&CMC monitors and manages work zones in coordination with other ITS systems. 

Construction and maintenance activities are tracked and coordinated with other systems, improving the quality and accuracy of information available regarding closures and other roadway construction and maintenance activities.
SoSR-1032

8.3.5 METR discrepancy handling system/centre (MDHS)

A METR discrepancy handling system accepts and manages reports of rule discrepancies received from METR users.  It will perform a numerical analysis of discrepancy count vs. time for any given rule, and when warranted, report to the METR regulation system those rules with sufficient discrepancy reports to justify further investigation.   

It may register its availability as a service, and its associated cyber information, with the SRADS.
SoSR-1042 NOTE:

A centre-based physical object that only depicts functional objects for the METR discrepancy handling system is called a "METR discrepancy handling centre".
SoSR-1033

8.3.6 METR distribution system (MDS)/centre (MDC)

A METR distribution system collects METR information from one or more METR regulation systems, and will verify that, holistically, the collective information is trustworthy for subsequent distribution to METR Users.  It will accept and respond to requests for METR information from METR consumer systems using the provided request criteria.

It may register its availability as a service, and its associated cyber information, with the SRADS.
SoSR-1041 NOTE:

A centre-based physical object that only depicts functional objects for the METR distribution system is called a "METR distribution centre".
SoSR-1034

8.3.7 METR input device (MID)

The METR input device represents any device used to enter METR information when done outside of a METR translation centre. For example, an emergency responder or maintenance and construction worker might need to activate a pre-authorized rule from the incident location. The MID might be realized by a smartphone, tablet, or the computer interface built into their vehicle.  
SoSR-1035

8.3.8 METR consumer system (MCS)

The METR consumer system receives guidance from a METR user system as to the current or projected needs for METR information.  It formulates and issues requests to the METR distribution system based on those needs and subsequently accepts and verifies the METR information received in response to each request.  It maintains and manages an information repository that enables the METR user system to continually have access to relevant METR information.

The METR consumer system can also transmit reports provided by the METR user system based on observations made while the vehicle traverses roadways and other transport facilities. Any METR consumer system is allowed to submit discrepancy reports (e.g., differences detected between METR rules and physical rules) to a discrepancy handling system. 

Authorized vehicles with specialized equipment can also provide notification of physical rules “discovered” directly to its associated MTS.
SoSR-1036

8.3.9 METR regulation system (MRS)

A METR regulation system enables the creation and verification of METR information, coordinating with other METR regulation systems as necessary to ensure trustworthiness.  It makes approved METR information available for distribution to one or more distribution systems.  It will verify that, holistically, the METR information it provides is consistent with rules from associated jurisdictional areas (e.g., the rules established by a city should be consistent with national rules). It then provides trustworthy information to distribution systems.  It coordinates the resolution of confirmed discrepancies with METR information, whether detected internally within the METR network, or as reported by METR consumer systems.
SoSR-1037

8.3.10 METR user system (MUS)

The MUS is typically either vehicle on-board equipment (OBE) or a personal information device (PID). For a full description of the OBE and PID, please refer to ARC-IT.   

The MUS combines METR information, as provided by the METR consumer system, with supporting data, as provided by other sources, to interpret which rules are currently active. The MUS provides its interpreted information to other on-board components to fulfil specific user needs. Typical uses include providing driver information and/or providing trustworthy information to driving automation systems. 

When the MUS detects discrepancies (e.g., a conflict between METR information and a physical rule detected by on-board sensors), it should notify the MMS so that it can transmit a METR discrepancy report. The MUS within an authorized vehicle can also notify the MMS of discovered rules for reporting to the associated MTS.
SoSR-1039

8.3.11 METR verification centre (MVC)

A METR verification centre provides independently verified METR information, as requested by a METR regulation system, with a primary focus on verification of METR rules with respect to the corresponding physical rule. 
SoSR-1040 NOTE:

A METR verification centre, while outside of the system boundary, is an integral part of the METR environment.
SoSR-1043

8.3.12 Non-METR distribution system (NMDS)

A non-METR distribution system will accept METR information from a METR regulation system but will disseminate METR information to its users using a non-conformant interface. 
SoSR-1046

8.3.13 Other METR discrepancy handling system (OMDHS)

Depending on deployments scenarios, a discrepancy report might be passed among multiple levels of discrepancy handling systems. For example, a vehicle might be configured to report all its detected discrepancies to an OEM-operated discrepancy handling system. This allows the OEM to become aware of potential problems with its discrepancy detection logic.

Once established thresholds are met, the OEM-operated discrepancy handling system might notify a discrepancy handling system operated by a jurisdictional entity. The jurisdictional entity system can then determine when the regulation system is notified of the reported discrepancies.
SoSR-1044

8.3.14 Other METR regulation system (OMRS)

A METR regulation system may exchange METR information with other METR regulation systems to enhance the trustworthiness, both in terms of accuracy and completeness, of METR information each respectively provides to METR users.
SoSR-1038

8.3.15 Specialized vehicle on-board equipment (SV OBE)

Within the context of METR, The SVOBE is typically realized as either an emergency vehicle OBE (EV OBE) or maintenance and construction vehicle OBE (MCV OBE). For a full description of the EVOBE and MCVOBE, please refer to ARC-IT.  

Within the context of METR, the SVOBE is a specialized MUS that is also able to act as a distribution system, and perhaps also as a streamlined regulation system. An SVOBE allows authorized agents (e.g., police officers, authorized construction workers) to issue METR rules, within their legal authorization, in a manner that promotes safety in response to current conditions.  
SoSR-1047

8.5.4.4 Internal information distribution

METR information that is approved can be sent to other METR and non-METR systems that are authorized to directly interact with the METR regulation system. Typically, METR information is forwarded to METR distribution systems to support trustworthy dissemination to METR users; the METR information may also be forwarded to non-METR distribution systems that convert trustworthy METR information into protocols that are not fully METR conformant (e.g., might not be fully trustworthy). For example, the METR information might be sent to smartphone navigation providers so that they have accurate data.   • there is a large volume of information to be distributed;
SoSR-946

  • most of this information is relatively static;
SoSR-1136

  • any of the information can potentially change with little notice;
SoSR-947

  • the information that needs to be distributed to any individual METR consumer system is a small subset of all METR information;
SoSR-948

  • individual METR users connect and disconnect at unpredictable times; and
SoSR-949

  • because METR consumer systems are typically a part of mobile devices, they might need to download a large amount of the relatively static data as they approach jurisdictional boundaries.

8.7.4.1 User reports of discrepancies

SoSR-764

8.7.4.2 Discrepancy report consolidation

The METR discrepancy handling system will receive discrepancy reports from multiple users over time, identifying the same discrepancy.  The handling system will track and consolidate information from all received reports for each rule discrepancy and based on local policies (e.g., sufficient reports have been received) will determine when to warrant a declaration of an identified rule discrepancy.   At that point it will generate and send a consolidated discrepancy report identifying the discrepancy and summarizing the characteristics of the received discrepancy reports (e.g., types and count of reporting vehicles). This report will be sent to a destination which might be another discrepancy handing system (e.g., an OEM sending data to a governmental handling system) but eventually, the consolidated reports are sent to the METR regulation system(s) that are responsible for the METR rules in question. 

METR regulation systems can also learn of discrepancies from other distribution or regulation systems that identify conflicts during their rule verification processes.
SoSR-765

8.7.4.3 Discrepancy disposition

When the METR regulation system becomes aware of a discrepancy, it will inform the appropriate rule translation agent of a reported discrepancy which should lead to an investigation and likely result in resolution with through corrective actions.  While investigating a discrepancy, a translator agent might wish to see additional details. If so, they can make a request and obtain the discrepancy report details.  To fulfil this task, discrepancy report details can be transferred between METR discrepancy handling systems.  

Fully resolving a discrepancy might require the coordination of multiple parties across multiple centres through use of the METR coordination flow.  If the discrepancy relates to a TCD (e.g., a missing TCD), the METR regulation system can use the TCD discrepancy status flow to request a correction to the TCD and to monitor the status of the correction.
SoSR-766

8.7.4.4 Suppression of confirmed discrepancies

Once a METR regulation system has been alerted to a discrepancy, it may wish to suppress further notices in one of two ways. If it still wants vehicles to gather data but for that data not to be submitted except on request, it can use discrepancy suppression info flow to prevent discrepancy handling systems from issuing consolidated discrepancy reports (but the newly gathered information is still available through the detailed flow). 

Alternatively, the regulation system can revise the associated rule meta-data to inform vehicles that a discrepancy with the rule is already known and that discrepancy reports should not be provided for the rule. This mechanism uses the two information distribution service packages defined in Clauses 8.5 and 8.6. 

Furthermore, the regulation system may wish to inform other discrepancy handling systems to correspondingly suppress further notices of those supressed discrepancies.

Each requirement in this document traces to, and is justified by, one or more user needs and/or constraints defined in the METR ConOps [1]. SoSR-955

11.2 Column Explanation

Each user need, constraint, and requirement is identified by its section number and its composite title. The composite title of an item consists of one or more component titles, each shown to a maximum length of 60 characters. The final component title is the title of the referenced item; if an item does not have a title, the text of the item serves as its title. If a component title is deemed to be not sufficiently unique and descriptive, it preceded by the title of its parent clause and separated with a pipe character ("|"). This process is repeated until the composite title provides a unique and descriptive title for the table.

Within the full listing, provided online, each item is also identified by a unique identifier, which identifies the source document (e.g., "ConOps") and a unique sequential identifier. The intent is that while future editions of the document might renumber section numbers, the unique identifiers will remain constant for a given item.
